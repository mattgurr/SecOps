# Career Paths Overview – Cohort 3 (Existing IT Professionals)

You already work in IT – that’s a huge head start. Moving into cybersecurity is less a “career change” and more a **career evolution**.

This guide shows common transition paths from IT roles into security, and how to turn your current work into proof you can handle a security role.

---

## 1. From Helpdesk / IT Support → SOC / Security Analyst

**Current roles:**
- Service Desk / Helpdesk
- Desktop Support
- Field Technician

**Target roles:**
- SOC Analyst (L1)
- Security Operations / Monitoring Analyst
- Junior Cybersecurity Analyst

**Why this path works:**
- You already handle tickets, triage, and user communication – core SOC skills.
- You know how systems behave in the real world and what “normal” looks like.

**Typical moves:**
- Start taking on security‑flavoured tasks: phishing triage, account compromise tickets, MFA onboarding, basic log checks.
- Shadow or assist the security team with low‑risk tasks.
- Move into a junior SOC / security analyst role once you’ve shown consistent value.

**Longer‑term path (0–5 years):**
- Helpdesk → SOC L1 → SOC L2 / Investigator → Detection Engineer / IR roles.

---

## 2. From Sysadmin / Network / Platform → Security Engineer / Cloud Security

**Current roles:**
- Sysadmin / Systems Engineer
- Network Engineer
- Cloud / Platform / SRE / DevOps Engineer

**Target roles:**
- Security Engineer
- Cloud Security Engineer
- Infrastructure / Platform Security

**Why this path works:**
- You understand how systems are built, deployed, and broken in production.
- Security teams need people who grasp real‑world constraints, not just theory.

**Typical moves:**
- Start owning security aspects of existing work:
  - Hardening baselines, patching, logging, backup/restore.
  - Network segmentation, firewall rules, VPNs.
  - Cloud IAM, guardrails, and security services.
- Lead small “security improvements” (e.g., better logging, secure templates).
- Transition into roles where security becomes your primary focus.

**Longer‑term path (0–5 years):**
- Sysadmin/Network/Cloud → Security/Cloud Security Engineer → Senior / Architect / Security Lead.

---

## 3. From Developer / DevOps → AppSec / Product Security

**Current roles:**
- Software Engineer / Developer
- DevOps / CI/CD Engineer
- SRE / Platform Engineer with strong code focus

**Target roles:**
- Application Security Engineer
- Product Security Engineer
- Secure SDLC / DevSecOps Engineer

**Why this path works:**
- You know code, pipelines, and release processes.
- You can talk to engineers in their language and embed security into existing workflows.

**Typical moves:**
- Introduce and own:
  - Dependency scanning, SAST/DAST, secret scanning in CI/CD.
  - Secure coding practices, threat modeling, and code review checklists.
- Partner with security teams on threat modeling or design reviews.
- Slide into formal AppSec / product security roles as you build trust and artifacts.

**Longer‑term path (0–5 years):**
- Dev/DevOps → AppSec / Product Security → Senior AppSec / Security Architect.

---

## 4. From IT / Technical PM / BA → GRC & Security Program Roles

**Current roles:**
- Technical Project Manager / Program Manager
- Business Analyst
- IT Service Management / Process Owner

**Target roles:**
- GRC Analyst
- Security Program / Compliance Manager
- Risk / Governance Lead

**Why this path works:**
- You already manage projects, stakeholders, and documentation.
- GRC needs people who can bridge business, technology, and regulation.

**Typical moves:**
- Take ownership of:
  - Security policy updates.
  - Risk registers for key systems.
  - Audit preparation and follow‑ups.
- Coordinate security initiatives and track their status.
- Move into dedicated GRC or security program roles as your portfolio of work grows.

---

## 5. Principles for Internal Transitions

Across all paths, similar principles apply:

1. **“Do security” in your current role first**  
   - Add security‑oriented tasks to your existing job: hardening, logging, awareness, basic detection, or policy work.

2. **Make the work visible**  
   - Document your improvements, dashboards, scripts, or runbooks.
   - Share results with your manager and relevant security leaders.

3. **Align with a realistic target role**  
   - Choose roles that build on your existing skills instead of ignoring them.
   - Use job ads and role descriptions to map skill gaps and close them deliberately.

4. **Use your 90‑day plan to prove fit**  
   - In this cohort, your 90‑day plan should produce:
     - A small security domain you “own”.
     - 1–2 internal mini‑projects with measurable impact.
     - A clear narrative for an internal or external security role move.

Your IT background is exactly what many security teams want: people who understand how things actually work in production, and can layer security on top without breaking everything.
